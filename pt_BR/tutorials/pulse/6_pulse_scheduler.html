Usando o Scheduler¶

O scheduler irá traduzir um QuantumCircuit em um Pulso Schedule, usando o portão e a medida de calibrações: uma descrição otimizada de nível-pulso de uma operação em determinados qubits.

Backends que são OpenPulse ativados geralmente terão calibrações definidas para medições e para cada um dos seus portões base. As calibrações também podem ser definidas ou atualizadas pelo usuário.

Uso básico¶

Para começar, construa um circuito quântico como você normalmente faria. Para o nosso exemplo abaixo, criamos um simples estado de Bell.

[1]:

from qiskit import QuantumCircuit

circ = QuantumCircuit(1, 1)
circ = QuantumCircuit(2, 2)
circ.h(0)
circ.cx(0, 1)
circ.measure([0, 1], [0, 1])

circ.draw()

[1]:

q_0: ┤ H ├──■──┤M├───
     └───┘┌─┴─┐└╥┘┌─┐
q_1: ─────┤ X ├─╫─┤M├
          └───┘ ║ └╥┘
c: 2/═══════════╩══╩═
                0  1

Vamos usar o backend simulado, FakeAlmaden, para demonstrar como usar o scheduler. Ele contém calibrações padrão para medida e para as suas portas base. A operação Hadamard não é uma dessas portas base, então usamos o transpiler para compilar nosso circuito de entrada para um circuito equivalente em termos das portas base do dispositivo.

[2]:

from qiskit import transpile, schedule as build_schedule
from qiskit.test.mock import FakeAlmaden

backend = FakeAlmaden()

transpiled_circ = transpile(circ, backend)  # Undefined Hadamard is replaced by U1
schedule = build_schedule(transpiled_circ, backend)

Vamos ver como está o nosso cronograma schedule construído a partir do circuito transpiled_circ. (Nós selecionamos um intervalo para plotar porque a medição e pulsos adquiridos são longos.)

[3]:

schedule.draw(plot_range=[0, 1700])

[3]:

Isso cobre o básico! Utilizamos o transpiler para reescrever o circuito nos termos dos portões base e, em seguida, utilizamos as calibrações padrão do backend para programar o circuito transpilado. A seguir vamos passar pelo agendamento com calibrações que nós mesmos construiremos.

Agendamento com definições personalizadas de portão¶

Usando o scheduler com Pulse, não ficamos limitados pelos portões base do dispositivo. Por exemplo, podemos definir uma calibração para o portão de Hadamard no qubit 0, que é usado pelo nosso circuito de estado Bell. Nosso Hadamard será um pulso Gaussiano. Na prática, o formato de pulso e seus parâmetros seriam otimizados através de uma série de experimentos Rabi (veja o ` Qiskit Textbook <https://qiskit.org/textbook/ch-quantum-hardware/calibrating-qubits-openpulse.html>` __ para uma visão geral).

[4]:

from qiskit.pulse import Schedule, Play, Gaussian, DriveChannel

h_q0 = Schedule(Play(Gaussian(duration=128, amp=0.045, sigma=8), DriveChannel(0)))

Adicionamos a calibração h_q0 para o InstructionScheduleMap existente do backend contendo a pesquisa de calibração de instrução. Ele tem que ser adicionado pelo nome do portão – aqui nós usamos o nome padrão do portão 'h' – e por qubit. Lembre-se de que um portão calibrado em um qubit não irá executar o mesmo portão em outro qubit.

[5]:

cals = backend.defaults().instruction_schedule_map
cals.add('h', 0, h_q0)
cals2 = backend.defaults().instruction_schedule_map

schedule = build_schedule(circ, backend, inst_map=cals)

Observe como dessa vez fomos capazes de agendar o circ, o qual não foi transpilado, então ele ainda tem o portão Hadamard. A nova calibração h_q0 que adicionamos foi usada para agendar circ.

Construir circuitos com portões não-padrão¶

Na seção anterior, adicionamos uma definição nativa para um portão padrão. Nesta seção, construímos um portão totalmente novo.

O primeiro passo é criar uma definição para ela como um Pulso Schedule. Abaixo, também demonstramos como estas agendas podem ser construídas para aceitar parâmetros.

[6]:

import numpy as np

from qiskit.pulse import Play, Waveform, ControlChannel
from qiskit.pulse.schedule import ParameterizedSchedule

def random_pulse(duration):
    if duration < 1:
        raise ValueError("Duration cannot be less than 1.")
    return Waveform(np.random.random(duration))

def random_2q_gate(x):
    sched = Schedule()
    sched += Play(random_pulse(x), DriveChannel(0))
    sched += Play(random_pulse(x), ControlChannel(0))
    sched += Play(random_pulse(x), DriveChannel(1))
    return sched

my_random_2q_gate = ParameterizedSchedule(random_2q_gate, parameters=['x'])

Chamamos a nossa operação de agendamento 'my_2q' e a definimos a partir dos qubits 0 e 1 conforme a adicionamos às calibrações.

[7]:

cals.add('my_2q', (0, 1), my_random_2q_gate)

Para preparar nosso circuito para usar esta definição, precisamos criar um novo Portão com o mesmo nome que definimos acima, 'my_2q'.

[8]:

from qiskit.circuit import Gate

my_2q = Gate('my_2q', num_qubits=2, params=[100])  # Here we pass `duration`, as 100

Agora, podemos construir nosso circuito. Quando usamos o novo portão, my_2q, nós o adicionamos ao circuito e, de outra forma, construímos o circuito normalmente.

[9]:

circ = QuantumCircuit(2, 2)
circ.h(0)
circ.append(my_2q, [0, 1])
circ.h(0)
circ.measure(0, 0)
circ.draw()

cals.remove('h', 0)

transpiled_circ = transpile(circ,
                            coupling_map=[[0, 1]],
                            basis_gates=backend.configuration().basis_gates + ['my_2q'])
transpiled_circ.draw()

Métodos do Scheduler¶

O scheduler pode seguir várias rotinas. Por padrão, segue uma regra o mais tarde possível (ALAP). Outro método de agendamento é o mais rápido possível, (ASAP). Para ambos os métodos, a schedule de saída é mínima: no caminho da operação de duração mais longa do circuito de entrada, o horário de início de cada operação é o momento final da operação seguinte. Os métodos determinam como agendar operações fora do caminho mais longo.

Isto fica claro através de um exemplo:

[12]:

circ = QuantumCircuit(2, 2)
circ.x(0)
circ.x(0)
circ.x(1)
circ.measure([0, 1], [0, 1])

[12]:

<qiskit.circuit.instructionset.InstructionSet at 0x138758470>

Para nosso backend simulado, essas operações :math: do portão X tem a mesma duração cada (duração = 160 dt). Para o agendamento ser mínimo, as duas operações :math: de X no qubit 0 serão agendadas back-to-back e o pulso de medição seguirá imediatamente o segundo desses pulsos.

ALAP irá escolher o último horário válido para agendar operações menos restritas, para que o :math: do portão X no qubit 1 execute em sincronia com o segundo :math: do portão X no bit 0.

[13]:

schedule = build_schedule(circ, backend,
                          method="as_late_as_possible")
schedule.draw(channels=[DriveChannel(0), DriveChannel(1)], scale=5)

[13]:

Por outro lado, como o nome sugere, o ASAP irá agendar as operações assim que seus recursos estiverem livres. Assim, :math: no portão X no qubit 1 será agendado em time=0, em sincronia com o primeiro :math: do portão X no qubit 0.

[14]:

schedule = build_schedule(circ, backend,
                          method="as_soon_as_possible")
schedule.draw(channels=[DriveChannel(0), DriveChannel(1)], scale=5)

[14]:

ALAP é o padrão porque permite que qubits permaneçam inativos durante o maior tempo possível. Neste caso, a diferença entre ALAP e ASAP pode ser negligenciável, mas no ALAP, o qubit 0 não tem tempo para decair do estado excitado antes da medição.
